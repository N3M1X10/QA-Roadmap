## AAA - arrange_act_assert
- Независимо от того, какой фреймворк используется, логика всегда сводится к трём шагам (паттерн AAA — Arrange, Act, Assert):
  - **Найди (Locate):** Мы даем скрипту команду найти конкретный элемент на странице (кнопку «Войти», поле ввода пароля).
  - **Сделай действие (Act):** Мы приказываем скрипту что-то сделать с найденным элементом (кликнуть, напечатать текст, навести мышку).
  - **Проверь (Assert):** Самое важное. Скрипт проверяет, совпадает ли реальный результат с ожидаемым. Например: «После клика на кнопку появился ли текст "Успешно"?». Если появился — тест зелёный (пройден). Если нет — красный (баг).

## DOM-дерево и Локаторы (Как скрипт видит страницу)

### 🌳 Что такое DOM-дерево (Document Object Model)
Когда браузер загружает страницу сайта, он превращает сырой HTML-код в интерактивную структуру — **DOM-дерево**.
- Представь это как структуру папок на компьютере или генеалогическое древо, где есть "родители" и "дети".
- Каждый элемент на странице (кнопка, картинка, текстовое поле) — это отдельный узел (нода) в этом дереве.
- У каждого элемента есть свой тег (например, `<button>`, `<input>`), текстовое содержимое и атрибуты (например, `id="submit"`, `class="primary-btn"`).

### 🎯 Что такое Локаторы (Locators)
Автотест не умеет смотреть на экран глазами. Чтобы кликнуть по кнопке или ввести текст, ему нужен **локатор** — точный адрес или описание элемента в DOM-дереве, по которому скрипт сможет его найти.

Современный стандарт Playwright — использовать **User-facing locators** (локаторы, ориентированные на то, что видит реальный пользователь). Они делают тесты стабильными: даже если разработчик перепишет внутренний код страницы, но кнопка визуально останется кнопкой, тест не упадет.

**Основные локаторы Playwright (примеры синтаксиса):**
1. Искать по роли и тексту на кнопке:
```javascript
page.getByRole('button', { name: 'Войти' })
```
2. Искать по подсказке (placeholder) внутри поля ввода:
```javascript
page.getByPlaceholder('Введите ваш телефон')
```
3. Искать по точному тексту на экране (например, для проверок):
```javascript
page.getByText('Заказ успешно создан')
```

### 🛠 Классические селекторы (CSS и XPath)
Если у элемента нет явного текста или роли, используются старые классические подходы:
* **CSS-селекторы:** Поиск по техническим атрибутам. Например, `.green-box` (найти элемент с классом green-box) или `#user-avatar` (найти элемент с уникальным ID user-avatar).
* **XPath:** Путь до элемента, прописанный через всю структуру DOM-дерева (например, `//div/ul/li[2]/button`). Мощный, но хрупкий инструмент — если структура страницы слегка изменится, локатор сломается. В Playwright используется как крайняя мера.

## Знакомство с Playwright и установка 🎭

### 🔥 Почему Playwright, а не Selenium?
Долгое время стандартом автоматизации был Selenium, но он работал медленно через специальный драйвер-посредник. Playwright устроен иначе:

* **Прямое управление:** Playwright подключается к браузерам напрямую через протокол WebSocket (как отладчик в Chrome). Из-за этого он работает молниеносно.
* **Авто-ожидание (Auto-waiting):** Больше никаких искусственных пауз типа "поспи 5 секунд, пока кнопка прогрузится". Playwright сам ждёт, пока элемент появится на экране, станет видимым и кликабельным.
* **Изоляция:** Каждый тест запускается в абсолютно чистом контексте (как вкладка инкогнито). Тесты не могут перепутать сессии или случайно залогиниться под чужими данными.

### 🛠 Как устроен проект (Базовая структура)
Когда мы разворачиваем Playwright, в папке проекта создаются ключевые элементы:
1. `playwright.config.js` — главный пульт управления. Здесь настраивается, в каких браузерах запускать тесты (Chromium, Firefox, WebKit), делать ли скриншоты при падении и куда складывать отчёты.
2. Папка `tests/` — место, где будут лежать наши файлы с автотестами (например, `login.spec.js`).

### 🚀 Быстрый старт: Команда установки
Чтобы развернуть Playwright в папке проекта, в терминале запускается всего одна команда:
```bash
npm init playwright@latest
```
Она скачает сам фреймворк, создаст базовую структуру папок и подтянет нужные версии браузеров прямо к тебе на компьютер.

## Анатомия и ключевые концепции Playwright 🎭

Чтобы писать автотесты, нужно понимать четыре главные сущности, из которых состоит фреймворк: **Browser**, **Context**, **Page** и **Тест-раннер**.

### 1. Иерархия управления (Браузер ➔ Контекст ➔ Страница)
Playwright не просто открывает окно Chrome. Он создаёт чёткую структуру:

* **Browser (Браузер):** Это сам запущенный движок (например, Chromium). Он запускается один раз для целой пачки тестов, чтобы не тратить оперативку компьютера на постоянный перезапуск тяжелого процесса.
* **Browser Context (Контекст):** Это изолированная сессия внутри браузера. Представь это как абсолютно чистую вкладку "Инкогнито". У каждого теста — свой контекст. Куки, кэш и сессии одного теста никак не могут протечь в другой. Тесты могут идти параллельно и не мешать друг другу.
* **Page (Страница):** Это конкретная вкладка внутри контекста. Именно с ней мы и работаем: даём команды перейти по URL, кликнуть или проверить текст.

### 2. Тест-раннер (Test Runner)
Это специальный модуль Playwright, который управляет запуском. Он ищет в проекте файлы с расширением `.spec.js` или `.test.js`, считывает их и решает, как их запускать.

Базовая конструкция любого теста в коде выглядит так:
```javascript
const { test, expect } = require('@playwright/test');

test('Название теста', async ({ page }) => {
  // Тут пишется логика теста
});
```
* `test` — функция, которая говорит раннеру: "Эй, внутри меня лежит тест, запусти его!".
* `async` и `await` — команды, которые заставляют JavaScript ждать выполнения действия (например, ждать, пока страница загрузится), прежде чем переходить к следующей строчке.
* `page` — та самая чистая вкладка, которую фреймворк автоматически создаёт и передаёт внутрь теста.
* `expect` — инструмент для финальных проверок (ассертов).
